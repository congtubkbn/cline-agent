Cline · agent execution loop

How the agent and model work — and how the run is logged

Traced from cline-log/1783615946293/ui_messages.json — a Qualcomm-case run on deepseek/deepseek-v4-flash. The model only reads and writes text; the harness does everything else and records every step as one event in a flat JSON array.

say — emit ask — needs user/UI model-authored content harness action (actor)

The turn loop — one cycle per model call

1

say : task User seeds the run

Prompt becomes event 0 at conversationHistoryIndex: -1, carrying modelInfo — model id + mode: "act".

2

say : checkpoint_created Snapshot before work

Harness saves a git-style lastCheckpointHash — a rollback point tied to the current turn index.

3

say : api_req_started Harness calls the model

Actor is the harness, not the model — this event records an API call the harness makes. It packs text: task + injected nudges + <environment_details> (open files, CWD, file tree, context-window %, mode), fires the request, then waits.

Written twice, same object. Created at send time with only tokensIn known. When the model finishes replying, the harness patches this same event in place and appends tokensOut / cacheReads / cost — the array entry you read is the post-patch, final state, not a snapshot from when it was sent.

4

say : reasoning · text · tool Model responds

One reply, split into events: reasoning (chain-of-thought) → text (user-facing) → tool, a structured call e.g. {"tool":"useSkill","path":"qualcomm-case-agent"}.

5

ask : command_output Harness runs it, gates side-effects

Tool executes (node -e …). Long stdout returns across several ask events. Result is injected into the next api_req_started — that's how the model "sees" the world.

↺ tool result becomes the next request → new turn, conversationHistoryIndex++ → back to step 3

Who does what

Every event in the file is written by the harness — the model never has file access. The split below is about who authored the content of an event, not who wrote it to disk.

The model

content author, not the writer

  • Reasons over the packed context
  • Picks a tool and emits its arguments
  • Writes user-facing text
  • Never touches disk, browser, or shell — never appends to the log itself

Cline (the harness)

the actor — does everything, records everything

  • Serializes the request, injects <environment_details> + nudges
  • Calls the model API — this is api_req_started; the model doesn't place this call itself, it's called at
  • Executes the tool call, captures stdout/stderr
  • Checkpoints each turn for rollback
  • Gates side-effects behind ask approvals
  • Writes every event — model-authored or its own — to ui_messages.json

How the log is stored

Same run can switch models mid-stream. …HistoryIndexMaps many UI events to one model turn. Multiple say/ask events share one index. actor ruleNo field states it directly — decode from subtype. task → user · reasoning/text/tool/command → model-authored (harness-written) · api_req_started/checkpoint_created → harness's own action · any ask → harness pausing for the user.
